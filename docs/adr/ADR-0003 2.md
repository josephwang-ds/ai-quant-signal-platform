# ADR-0003: Apply Clean Architecture Through Vertical Slices

- **Status:** Accepted
- **Date:** 2026-07-13
- **Decision owners:** Architecture maintainers
- **Scope:** Source organization and dependency direction

## Context

The platform needs strict separation between business authority and volatile technologies while still delivering use cases incrementally. Pure horizontal layering tends to scatter a change across unrelated folders; unstructured feature folders tend to mix domain, transport, and persistence concerns.

## Decision

Use Clean Architecture inside each bounded-context module and deliver work as vertical slices.

Dependencies point inward: Presentation depends on Application; Application depends on Domain; Infrastructure implements ports defined inward; Bootstrap composes concrete dependencies. A vertical slice contains the smallest complete path for one named use case while preserving those layers.

## Consequences

### Positive

- Business rules remain framework-independent and testable.
- Changes are organized around outcomes rather than technical layers alone.
- Adapters can be replaced without rewriting domain behavior.

### Trade-offs

- Simple use cases have more explicit contracts.
- Shared abstractions must be earned to avoid premature frameworks.
- Reviewers must check both slice cohesion and layer boundaries.

## Alternatives considered

- **Traditional controller/service/repository layering:** rejected because use cases become diffuse and service layers accumulate business rules.
- **Framework-first modules:** rejected because domain code would depend on delivery technology.
- **Unlayered feature folders:** rejected because boundary discipline would be implicit.

## Compliance

- Domain imports no FastAPI, SQLAlchemy, database client, scheduler, or LLM SDK.
- Application imports ports, not concrete adapters.
- Routes and jobs invoke Application use cases only.
- Slice tests prove domain behavior and boundary behavior separately.
