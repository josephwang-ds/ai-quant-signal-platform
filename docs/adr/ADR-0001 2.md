# ADR-0001: Adopt a Modular Monolith as the Default Runtime

- **Status:** Accepted
- **Date:** 2026-07-13
- **Decision owners:** Architecture maintainers
- **Scope:** Runtime topology

## Context

The platform requires strong domain boundaries and independent module evolution, but it does not yet have operational evidence that distributed deployment is worth its consistency, observability, and delivery cost. Research workflows also coordinate multiple domain capabilities while preserving a single governed history.

## Decision

Build and operate the platform as a modular monolith. Research, Validation, Governance, Portfolio, and Market Intelligence are explicit bounded-context modules deployed as one application by default. Modules own their behavior and data access, expose deliberate contracts, and communicate through Application use cases or domain-event handlers.

The architecture does not permit a “distributed monolith” inside one process: modules may not reach into another module’s repositories, tables, or internal types.

## Consequences

### Positive

- Atomic workflows and local development remain straightforward.
- Architecture boundaries can mature before deployment boundaries.
- Operational overhead stays proportional to the product’s current scale.

### Trade-offs

- Automated boundary checks and disciplined review are required.
- A noisy workload can affect the shared runtime until isolated intentionally.
- Independent deployment is deferred.

## Alternatives considered

- **Microservices now:** rejected because topology would precede proven scaling and team needs.
- **Layered monolith without modules:** rejected because domain ownership would erode.
- **Serverless function per use case:** rejected as the default because shared workflows and observability would fragment.

## Compliance

- Each capability belongs to one bounded-context module.
- Cross-context access uses public contracts or events.
- Extraction requires an ADR supported by operational evidence.
